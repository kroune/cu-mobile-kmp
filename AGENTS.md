Важно чтобы код в проекте не просто работал, но и **хорошо поддерживался**

## Правила разработки

- Используй современный Kotlin и Compose.
- Переиспользуй функции, выноси общую логику в отдельные функции, используй OOP.
- При создании/редактировании файлов проверяй соответствие архитектуре проекта.
- В одной строчке кода не должно быть сложной логики — это ухудшает читаемость.
- Инвалидируй кэши только в самых крайних случаях. Для перезапуска тестов используй `--rerun`.
- В директории `memorybank/` находится информация, сохранённая ИИ в предыдущих сессиях. Сохраняй
  туда важную информацию.
- Перед началом работы читай `memorybank/ARCHITECTURE.md`
- Если информации в `ARCHITECTURE.md` не хватает — читай нужные файлы, затем после обновляй `ARCHITECTURE.md` так, чтобы в следующей сессии не пришлось читать
  все файлы.
- Если `ARCHITECTURE.md` становится слишком большим — декомпозируй его на несколько файлов по разным аспектам (например по компонентам).
- Если неуверен в синтаксисе библиотеки — используй MCP DeepWiki (или context7).
- Запланированные к реализации фичи (с приоритетами) — в `memorybank/remaining-work-plan.md`. После выполнения задачи обновляй этот файл.
- Если заметил какие-то спорные моменты, проблемы в коде, которые не относятся к текущим изменениям и т.п. - записывай информацию в
  `memorybank/POTENTIALLY_BAD.md`
- Обязательно делай логирование ошибок, никогда их не съедай
- Это нормально добавлять дебаг логи для отладки, надо будет просто убрать логирование когда проблема будет решена
- важно чтобы у приложения был хороший performance
