# Краткое описание проекта:

Цель проекта - написание kmp приложения для лмс ЦУ (центральный университет). Поддержка android,
ios. Предпочтительно шарить большую часть кода между двумя платформами.

Сейчас уже есть рабочая версия на flutter (https://github.com/cu-3rd-party/lms-mobile), он уже
проиндексирован deepwiki, а так же установлен в /home/olowo/StudioProjects/lms-mobile. Ты должен
аккуратно сделать fork, не сломав функционал. В случае сложностей с запуском, недоступностью mcp или если надо войти в приложение - обращайся ко мне.
Так же тебе стоит проверять наличие изменений в ui через ./gradlew verifyRoborazzi и проверять что они все выглядят правильно. Перезаписывай screenshot-ы только если ты проверил все изменения и убедился что они работают корректно. Текущие скриншоты находятся в androidApp/screenshots

Важно чтобы код в проекте не просто работал, но и **хорошо поддерживался**

## Правила разработки

- Используй современный Kotlin и Compose.
- Переиспользуй функции, выноси общую логику в отдельные функции или используй OOP.
- При создании/редактировании файлов проверяй соответствие архитектуре проекта.
- В одной строчке кода не должно быть сложной логики — это ухудшает читаемость.
- При создании UI пиши `@Preview`-функции (в них нормально делать suppress magicNumber).
- Используй `detektAll`. Фикси нарушения, не делай suppress
- Инвалидируй кэши только в самых крайних случаях. Для перезапуска тестов используй `--rerun`.
- В директории `memorybank/` находится информация, сохранённая ИИ в предыдущих сессиях. Сохраняй
  туда важную информацию.
- Перед началом работы читай `memorybank/ARCHITECTURE.md`
- Если информации в `ARCHITECTURE.md` не хватает — читай нужные файлы, затем обнови `ARCHITECTURE.md`.
- После изменений обновляй `ARCHITECTURE.md` так, чтобы в следующей сессии не пришлось читать
  все файлы.
- Если `ARCHITECTURE.md` становится слишком большим — декомпозируй его.
- Если неуверен в синтаксисе библиотеки — используй MCP DeepWiki (или context7).
- Запланированные к реализации фичи (с приоритетами) — в `memorybank/remaining-work-plan.md`. После выполнения задачи обновляй этот файл.
- Если заметил какие-то спорные моменты, которые требуют моего решения или сложную проблему - записывай информацию в
  `memorybank/HUMAN_NEEDED.md`
- Обязательно делай логирование ошибок, никогда их не съедай
- Если фича большая (требует >15 файлов или expect/actual пары, сложную оркестрацию) — **не начинай её в текущей сессии**. Запиши детали реализации в `memorybank/ARCHITECTURE.md` и скажи пользователю: «Запусти новую сессию для этой фичи». Предпочти сохранить контекст и передать чистый старт, чем делать большую фичу при забитом контексте. Если сессия пустая и пользователь сказал делать фичу - тогда делай ее
- Это нормально добавлять дебаг логи для отладки, надо будет просто убрать логирование когда проблема будет решена
